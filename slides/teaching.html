Validation & linters
02
Three kinds of check
All "linting" is not the same. Sort every tool into one of three groups — it tells you how hard to enforce it.
GROUP 1

Syntax

Does the file even parse? Cheap, fast, hard to argue with. A trailing comma in project.json means the gateway can't load your resource.

yamllint · docker compose config
actionlint · ops/validate.sh
GROUP 2

Best practice

Will this bite you later? Not a syntax error — a "works today, breaks the moment someone renames something" warning. The most valuable group: it encodes operational wisdom.

shellcheck (unquoted $VAR)
ign-lint (brittle refs, runaway polls)
GROUP 3

Style

Is it consistent? Useful in a team — consistency reduces cognitive load — but the most fiddly. Tune or disable rules, as long as the team agrees and the config is checked in.

line-length · naming conventions
(PascalCase components…)
The rule of thumb

Enable groups 1 and 2 everywhere; treat group 3 as opt-in. And the config lives in the repo — .yamllint.yml, rule_config.json — so the team agreement is version-controlled and enforced, not tribal knowledge.

Validation & linters · the flagship
04
ign-lint — the linter that understands Ignition
The other tools are generic. ign-lint parses Perspective view.json, walks the component tree, and checks the Python embedded in views — without a running gateway. Configured by rule_config.json in the repo root.
NamePatternRulecomponents → PascalCaseerror · properties → camelCasewarn · handlers → kebab-casewarn · methods → snake_caseerror
PollingIntervalRulefloor on binding poll rates — 1000 ms in this lab. now(250) is a finding.
BadComponentReferenceRuleflags brittle traversal — .getSibling(), .getParent(), relative pathserror
ComponentReferenceValidationRuledo relative references actually resolve? Validates expressions, bindings, scripts.
PylintScriptRuleruns pylint over the Python inside views, mapping categories to error/warning.
UnusedCustomPropertiesRulecustom properties nothing reads.
ExcessiveContextDataRuleoversized context data — big arrays, deep nesting, too many data points.
# install (needs Python 3.10+)
pip install ign-lint==0.6.1

# run — same command locally and in CI
ign-lint --config rule_config.json \
  --files "projects/**/view.json"

Validation & linters · the backstop
05
ops/validate.sh — the backstop, and the limits
ign-lint goes deep on views; validate.sh goes broad on everything. Nothing but python3, about a second, exit 0 or 1.
# what it checks — the whole thing:
# 1. every *.json under projects/ is valid JSON
# 2. every code.py parses as Python 3
ops/validate.sh
→ ✓ all project files valid   (exit 0)

# after a hand-edit leaves a trailing comma…
ops/validate.sh
→ ✗ invalid JSON: projects/lab-project/project.json
→ ✗ validation failed          (exit 1)
Honest fine print

The gateway runs Jython 2.7; this is a fast Python-3 parse check, not a Jython validator. Write Python-3-parseable syntax and you're fine — the lab's scripts already are.

LINTING HELPS WHEN…

· it catches the bug every dev would hit
· it's fast enough to run on save (or pre-commit)
· the config matches your team's reality, not a pasted default

…AND HURTS WHEN

· it flags style as errors and blocks PRs
· it's slower than the dev loop
· the config drifts from reality and nobody agrees with it anymore

the cure: tune the checked-in config — don't disable the linter

GitHub Actions · triggers
06
Triggers — the on: block
Events start workflows. This lab uses two, plus path filters so docs-only PRs don't waste a runner.
# the lab's ci.yml — real triggers
on:
  pull_request:          # every PR update
    paths:               # …that touches code we check
      - "projects/**"
      - "ops/**"
      - "docker-compose.yml"
      - ".github/workflows/**"
      - ".yamllint.yml"
      - "rule_config.json"
  push:
    branches: [main]     # backstop after merge
  • pull_request — runs on open and on every push to the PR branch. The workhorse for checks.
  • push: branches: [main] — the backstop: verifies main itself stays green after merges and direct pushes.
  • paths: — a README-only PR doesn't spin up two VMs to lint files it never touched. The workflow is skipped, not passed — you'll verify the difference in the lab.
  • Others you'll meet: workflow_dispatch (a manual Run button — the runner demo uses it), schedule (cron), release.
Filter discipline

Too narrow (one file) and view changes sneak past. Too broad ("**") and the filter is pointless. Cover the directories the checks actually read — including the configs like rule_config.json.

GitHub Actions · the runner model
Where does it actually run?

Every job gets a fresh, ephemeral virtual machine. Jobs run in parallel and share nothing.

event You push to a PR branch → GitHub matches it against on: → schedules the workflow's jobs.
job: lintfresh VM · ubuntu-latest
checkout — clone your repo into the empty VM1
setup-python "3.12"2
pip install … → yamllint → actionlint → shellcheck → ign-lint3…7
job ends → VM destroyed. Nothing survives.
job: validateanother fresh VM
checkout — its own clone; it saw nothing of lint's VM1
ops/validate.sh → exit 0 or 1 = the job's result2
job ends → VM destroyed. Nothing survives.
It starts with an event — a push to a PR branch.
GitHub Actions · the lab's workflow
07
Reading the lab's ci.yml
This is the file you'll build from scratch in the I-do and finish in the lab. Two independent jobs — the Part 1 linters, and the validator.
name: CI
on:
  pull_request:
    paths: ["projects/**", "ops/**", …]
  push:
    branches: [main]
permissions:
  contents: read          # least privilege
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"   # quoted!
      - run: pip install yamllint==1.35.1 ign-lint==0.6.1
      - run: yamllint -c .yamllint.yml .
      - uses: raven-actions/actionlint@v2
      - run: shellcheck ops/*.sh   # apt-get install first
      - run: ign-lint --config rule_config.json --files "projects/**/view.json"
      - run: docker compose config -q
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ops/validate.sh
  • Two jobs, two VMs, parallel. Independent failures: a shell bug fails lint; broken JSON fails validate.
  • Steps are sequential — a failing step skips the rest of its job and fails it. (continue-on-error and needs: exist for when you want different behaviour.)
  • "3.12" is quoted because unquoted YAML coerces it to the float 3.1. A classic. And ign-lint needs Python 3.10+, so the pin matters.
  • The --files glob is quoted so the runner's shell doesn't expand it before ign-lint sees it.
  • No reusable workflows here — deliberately. You see what's inside before you ever call one.

GitHub Actions · trust
08
Permissions, GITHUB_TOKEN & secrets
A workflow is code that runs with credentials. Treat those credentials like production access — because one day they will be.
  • permissions: contents: read — least privilege at the workflow level. The default GITHUB_TOKEN grant is broader than a lint job ever needs; our jobs only read the repo, so that's all they get.
  • GITHUB_TOKEN — auto-provisioned per job, scoped to this repo, expires when the job ends. You never create it, rotate it, or store it.
  • Secrets — encrypted at rest, injected on demand, and masked as *** in logs. In the I-do we add an EXAMPLE_SECRET, echo it, and watch GitHub censor it live.
  • Variables — the unencrypted sibling. Fine for a gateway URL; never for a password. If it would hurt in a screenshot, it's a secret.
Foreshadowing

Masking is a display feature, not a security boundary — a malicious workflow can still exfiltrate a secret it has access to. That's why permissions and triggers matter, and why Day 4 spends a whole session on secrets management done properly.

GitHub Actions · the gate
09
Required checks — CI grows teeth
A green workflow is information. A required status check is enforcement: branch protection on main disables the merge button until lint and validate pass.
  • Settings → Branches → protect main: require a PR before merging, and require the lint and validate checks to pass.
  • The shift in language: from "please remember to run the linters" to "you cannot merge until they pass." The safety net stops depending on memory.
  • You'll prove it in the lab: plant the broken state, open a PR, and watch GitHub physically block the merge — then fix, re-push, and merge.
It's a team contract

A required check is a contract with the team, not just a setting. Who decides what's blocking? Who maintains CI when it's flaky? A check the team routinely overrides is worse than no check — it trains everyone to ignore red.

Also in this club

Required reviews (Day 1's PR discipline) and required checks compose: human judgement and machine verification, both enforced by the same branch-protection rule.

The deployment environment
10
The deployment environment — one gateway, wired for CI
Everything so far ran gateway-free — but the environment those files deploy to is in the repo too. docker-compose.yml defines one Ignition 8.3 gateway; ops/ gives it a lifecycle. Two mounts explain the whole design.
  • Named volume ignition-data — the gateway's own config + runtime state, self-generated on first boot. It never lands in the repo, so you never touch gateway config here (that's Lab 04).
  • Bind mount ./projects — the one thing edited from the repo. The files ign-lint and validate.sh check are the files the gateway runs — no packaging step for drift to hide in.
  • Edits aren't auto-detected — a running gateway won't see changes to the bind-mounted files until you trigger a project scan (ops/scan.sh, needs an API key in .env) or docker compose restart.
  • The bridge to Chapter 3: this gateway answers on localhost:8088. A production gateway sits behind a firewall — and deploying into it (Labs 04–05) is exactly when CI needs a machine that can reach it.
# docker-compose.yml — the two mounts that matter
volumes:
  - ignition-data:/usr/local/bin/ignition/data        # gateway-owned
  - ./projects:/usr/local/bin/ignition/data/projects  # the repo's files

# its lifecycle — ops/
ops/setup.sh      # boot + wait for RUNNING → localhost:8088
ops/scan.sh       # push project-file edits to the live gateway
ops/teardown.sh   # stop · --volumes wipes gateway state
Why CI never boots it

Every check in ci.yml validates this environment without starting it: docker compose config -q checks the environment's definition; ign-lint and ops/validate.sh check the project files it serves. Booting a gateway per PR would be slow and stateful — the checks are fast precisely because they don't.

Chapter 3
What if CI can't reach your gateway?
GitHub-hosted runners live in GitHub's cloud. Your production gateway lives behind a firewall, on a customer VPN, next to a PLC network. Sooner or later, CI has to come to you.

Self-hosted runners · architecture
The runner comes to the network — not the other way

A runner is a lightweight agent that polls GitHub outbound for jobs. There is no inbound connection from GitHub — nothing to open on the firewall.

GitHub's cloud
Actions service — queues jobs, stores logs
GitHub-hosted runners (ubuntu-latest…)
fine for lint + validate — no plant access needed
firewall / VPN
Your network (plant / customer site)
self-hosted runner — polls outbound: "any jobs for me?" → executes → returns logs
Ignition gateway(s) — the deploy target GitHub-hosted can never reach
PLC network · historian DB · internal artifact registry
  • When you need one: ① network isolation — the big one for Ignition; ② compliance / data residency; ③ real hardware (a physical PLC, an OPC-UA server); ④ cost at very high volume — rarely, don't optimise prematurely.
  • When you don't: everything in today's lab. Lint and validate run on a standard image — use GitHub-hosted and move on. A runner host is a machine you now own, patch, and monitor.

Self-hosted runners · security
11
The one rule — and the guardrails
A self-hosted runner executes arbitrary workflow code inside your network, with the host's filesystem, network reach, and cached credentials. Treat it as highly trusted infrastructure.
Never do this

Never attach a self-hosted runner to a public repo that accepts fork PRs. A malicious PR edits the workflow, the runner dutifully executes it — exfiltrated secrets, a backdoor, a pivot into the PLC network. This is not theoretical.

# the demo you'll see in the I-do:
# ephemeral Docker runner, one job, then gone
docker run -d --rm --name lab03-runner \
  -e REPO_URL="https://github.com/<you>/<repo>" \
  -e RUNNER_TOKEN="$RUNNER_TOKEN" \
  -e LABELS="self-hosted,local-lab03" \
  -e EPHEMERAL=true \
  myoung34/github-runner:latest
  • Ephemeral runners — destroyed after every job; a compromised job leaves nothing behind.
  • Private repos only, restricted workflow triggers, approval required for first-time contributors.
  • Network-isolate the runner — its own subnet, only the outbound access it needs. Never production credentials on the host; short-lived tokens via OIDC.
  • The runner only has what you put on it. No preinstalled shellcheck, no Python — the "works on ubuntu-latest" assumption dies here. You'll see this live in the demo.
  • Where it sits for Ignition: a dedicated build host with network access to the gateway — not on the gateway server itself. Hands-on in Labs 04–05.
